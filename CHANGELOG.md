# PPBears SaaS 更新日誌

本檔案將記錄此專案所有值得注意的更新與變動。

## [v0.5.28] - 2026-04-10

- `test-connection`：新增 `proxy_url` / `proxy_secret` 狀態欄位，可確認 Render env var 是否已正確載入

## [v0.5.27] - 2026-04-10
### 🐛 修正：訂單查詢持續失敗（Hostinger Imunify360 封鎖 Render IP）
- **根本原因**：Hostinger Imunify360 WAF 將 Render 伺服器 IP（`74.220.49.248`）列為黑名單，導致後端直接呼叫 WooCommerce REST API 回傳 403 Forbidden。Hostinger IP Manager 白名單**無法覆蓋** Imunify360 規則；Hostinger 客服確認因信譽問題無法解除封鎖。
- **解決方案（PHP Proxy）**：在 Hostinger 伺服器上部署 `wc-proxy.php`（`public_html/wc-proxy.php`），由 PHP 腳本在**伺服器本機**呼叫 WooCommerce API（請求來源 IP 為 `127.0.0.1`，不受 Imunify360 外部 IP 過濾）。Render 後端改透過此代理呼叫，並以 `X-Proxy-Secret` Header 驗證身份防止濫用。
- **修改檔案**：
  - 新增 `hostinger/wc-proxy.php`（需手動上傳至 Hostinger `public_html/`）
  - `woocommerce.service.ts`：新增 `buildRequest()` 方法，偵測環境變數 `WOO_PROXY_URL` / `WOO_PROXY_SECRET`，有值時自動走代理路徑，無值時維持原有直連邏輯（向下相容）
- **需設定 Render 環境變數**：
  - `WOO_PROXY_URL` = `https://www.ppbears.com/wc-proxy.php`
  - `WOO_PROXY_SECRET` = `ppbx_8f3a2c9d7b1e4f6a0e5d2c8b`

## [v0.5.26] - 2026-04-10
### 🐛 修正：客戶查詢訂單回覆「查無訂單」
- **根本原因**：`woocommerce.service.ts`（訂單查詢）的 `getCredentials()` 在建構 API URL 時，直接使用資料庫儲存的 `woo_base_url`（非 www），沒有自動補上 `www.` 前綴，導致請求被 Hostinger Imunify360 防火牆在 TCP 層級封鎖，WooCommerce API 呼叫失敗並 silently 回傳 `null`，最終顯示「查無訂單」。`quick-order.service.ts` 與 `product.service.ts` 已在 v0.5.22 修正此問題，但 `woocommerce.service.ts` 當時被遺漏。
- **修復方式**：在 `woocommerce.service.ts` 的 `getCredentials()` 加入與其他服務一致的 `www.` 自動補全邏輯（`rawBaseUrl.replace(/^(https?:\/\/)(?!www\.)/i, '$1www.')`），確保訂單查詢 API 請求能穿透 Hostinger 防火牆。

## [v0.5.25] - 2026-04-02
### 🐛 修正：24小時後真人接管自動還給 AI，但 UI 仍顯示「真人接管」狀態
- **根本原因**：`cleanupExpired()` 方法雖存在但**從未被呼叫**。Render 後端無狀態環境沒有 Cron，導致過期 session 的 `released_at` 永遠是 `null`，`conversations.status` 永遠停在 `live_agent`。機器人靠 `expires_at > now` 正確還給了 AI，但 UI 卻永遠看到「真人接管」。
- **修復方式**：在 `listConversations()` 中加入即時過期偵測：每次 API 被呼叫時（UI 每 5 秒輪詢），自動檢查 `live_agent` 對話的 `expires_at`，若已過期則即時回傳 `active` 狀態給 UI，並 fire-and-forget 更新資料庫。

## [v0.5.23] - 2026-04-02
### 🐛 修正：OPPO Reno13 5G 等品牌搜尋失敗 + AI 幻覺「我們只做三星」
- **根本原因 1（搜尋失敗）**：`reno13` 沒有被斷詞拆開，`5g` 作為 anchor token 干擾搜尋結果，且 `oppo` 不在 `BRAND_MAP` 導致品牌過濾無法啟動。
- **根本原因 2（AI 幻覺）**：商品搜尋失敗後，not-found guard prompt 只說「告知找不到」，AI 自行用知識庫中的敘述（三星相關）補充「原因」，捏造出「我們目前專注三星手機殼」等完全錯誤的說法。
- **修復方式 1**：`product.service.ts` Step 2d 新增 reno/oppo/vivo/realme 等品牌的數字前斷詞（`reno13` → `reno 13`）；Step 2e 新增 5G/4G/3G 等通訊制式雜訊過濾；`BRAND_MAP` 加入 OPPO/Vivo/Realme/Sony/Google/Motorola 英文品牌自身條目，確保 anchorGroups 能正確施加品牌過濾。
- **修復方式 2**：`orchestrator.ts` not-found guard 強化，明確插入「**嚴格禁止說我們只做某品牌手機殼**，PPBears 提供多品牌服務，系統索引找不到不代表沒有此商品」的規則，並要求 AI 直接引導客戶輸入真人。

## [v0.5.22] - 2026-04-02
### 🐛 修正：快速開單（ppbears888 開單）失敗
- **根本原因**：`quick-order.service.ts` 在建構 WooCommerce POST API URL 時，直接使用資料庫儲存的 `woo_base_url`（非 www），沒有自動補上 `www.` 前綴，導致請求被 Hostinger Imunify360 防火牆在 TCP 層級封鎖，建立商品的 POST 請求回傳失敗，LINE 端顯示「⚠️ 開單失敗」。
- **修復方式**：在 `getSettings()` 中加入與 `product.service.ts` 完全一致的 `www.` 自動補全邏輯（`rawWooBaseUrl.replace(/^(https?:\/\/)(?!www\.)/i, '$1www.')`），確保快速開單 POST 請求與商品同步走同一條穿透路徑。

## [v0.5.16] - 2026-04-01
### 🐛 修正：AI 輸出「[產品索引查到的連結]」佔位詞而非真實 URL
- **根本原因 1**：`formatProductsAsAiContext` 使用 `[1]` 括號數字格式，AI 模型將其視為參考引用樣板，回覆時自創 `[產品索引查到的連結]` 佔位詞代替真實 URL。
- **根本原因 2**：若資料庫中商品的 `url` 欄位為空字串，格式輸出為 `購買連結: `（後面為空），AI 看到沒有 URL 後自動補一個假佔位詞。
- **修復方式 1**：在 `formatProductsAsAiContext` 加入 URL 合法性過濾，只有 `url.startsWith('http')` 的商品才進入格式化；若全部過濾後無商品，回傳空值讓後端進入「找不到」流程。
- **修復方式 2**：將格式從 `[1] 商品名稱: X | 購買連結: URL` 改為純文字多行格式，消除括號數字引發的樣板聯想。
- **修復方式 3**：明確在 `[回覆規則]` 中加入「絕對不可替換成 `[產品索引查到的連結]`、`[連結]` 等任何佔位詞，必須原文複製完整 URL」。

## [v0.5.15] - 2026-04-01
### 🐛 修正：確認詞後未繼承產品搜尋（S23U → 確認 → 沒給連結）
- **根本原因**：客戶說「S23U」，AI 反問「您是指 S23 Ultra 嗎？」，客戶回「是的 確認」，系統判定「是的 確認」不是產品查詢意圖（沒有品牌/數字/英文），直接跳過產品索引，AI 只能給通用回答。
- **修復方式**：新增「確認詞繼承搜尋」機制。當偵測到短確認詞（是、對、確認、ok、好的⋯）且當前訊息本身不是產品查詢時，往前查找最近一條用戶的產品查詢訊息，自動以該訊息作為搜尋關鍵字重新執行產品搜尋，確保確認後能立即拿到正確產品連結。

## [v0.5.14] - 2026-04-01
### 🐛 修正：型號縮寫雙向匹配（S23 U ↔ S23 Ultra 視為同一產品）
- **根本原因**：產品資料庫中商品名稱記錄為「S23 U」（縮寫），但客戶輸入「S23 Ultra」時，搜尋引擎只展開「u→ultra」，沒有展開「ultra→u」，導致用全名搜不到縮寫商品。
- **修復方式**：將 `ALIAS_MAP` 從單向改為完整**雙向**對照表，`ultra` 同時也搜 `u`，`promax` 同時也搜 `pm` / `pro max`。不論客戶打縮寫還是全名，都能命中資料庫商品，彻底解決 S23U / S23 U / S23 Ultra 三者互相無法匹配的問題。

## [v0.5.13] - 2026-04-01
### 🐛 修正：AI 搜尋產品時被「磨砂殼/透明殼」等材質詞干擾導致找不到產品連結
- **根本原因**：`extractSearchKeyword()` 採用「黑名單過濾」策略，只移除特定雜詞，卻未考慮到「磨砂殼」「透明殼」等材質描述詞。這類 token 殘留進入 `searchProducts()` 的 **AND-of-ORs** 查詢，要求商品名稱也必須包含「磨砂殼」，但資料庫商品名稱並不含此字，導致 AND 條件強制歸零，找不到任何產品，AI 因此改用通用教學連結應付。
- **修復方式**：將 `extractSearchKeyword()` 從「黑名單刪除法」完全改寫為**「正向抽取法（Whitelist Extraction）」**：只保留英文字母、數字、以及已知中文品牌名（三星、蘋果、小米等），其他所有中文字（材質、動作詞、形容詞）一律丟棄。例如「三星 s23 ultra 磨砂殼」→抽取為「三星 s23 ultra」，乾淨且精準。

## [v0.5.12] - 2026-03-31
### ✨ 增強：跨回合互動的「上下文繼承搜尋」與「找不到引導真人」機制
- **修改原因**：當使用者先打「17」，AI 丟出反問「請問是 Apple 17 還是 Xiaomi 17？」，使用者此時回覆「apple」，系統由於每次搜尋都是獨立的，只拿「apple」去搜資料庫，導致完全丟失了上一句的型號「17」，進而回傳了一堆不相干的 iPhone 結果給 AI，讓 AI 誤判說沒有賣 iPhone 17。
- **修復方式 (上下文繼承)**：現在搜尋引擎加入**短句語境推斷 (Contextual Injection)**。如果客戶的回覆非常短（如單一個品牌名），系統會主動抓取客戶的上一句話拼裝起來（例如變成 `17 apple`）再送進搜尋引擎，確保跨回合問答時不會失憶。
- **修復方式 (體驗升級)**：進一步更新了 AI 提示詞。如果客戶明確表示「沒有」這個型號，或者系統多次來回仍找不到商品，AI 會主動建議客戶：「您可以輸入『真人』，我將為您轉交專員尋找唷！」來提升客製化商店的服務溫度。

## [v0.5.9] - 2026-03-31
### ✨ 重構：全面泛化意圖偵測與實作「跨品牌模糊反問」機制
- **修改原因**：原有的意圖偵測只硬背了 Pro/Max/Ultra 等後綴，無法覆蓋 OPPO/VIVO/ROG 等其他 Android 廠牌千奇百怪的型號（如 Reno 12, X100, V30）。且針對極度模糊的數字（如「17」），AI 容易在收到跨品牌結果時直接當機胡說。
- **修復方式**：
  1. 廢除死背名單的正規表達式，擴充 10 餘家主流品牌至意圖守門員白名單，並改為抓取泛化特徵（例如 `1-4字母加上1-3數字`、`長度小於15字的純數字` 等）。
  2. 新增「跨品牌衝突攔截機制」：在給 AI 組合參考提示詞時，系統會掃描返回的商品是否同時隸屬兩個差異極大的品牌（如 Apple 與 Xiaomi）。如果是，系統會攔截商品清單，改為傳送系統指令強迫 AI 回覆：「請問您是指 iPhone 的款式，還是小米或其他廠牌呢？」避免 AI 亂給連結！

## [v0.5.8] - 2026-03-31
### 🐛 修正：連體嬰查詢（如 17PROMAX）直接被意圖過濾器丟掉的問題
- **根本原因**：上一版（v0.5.7）雖然完美重寫了**底層產品搜尋分析器（Tokenizer）**，但系統大門口的「**意圖偵測守門員（Intent Detector）**」卻有個 Bug。守門員的正則表達式 `\b\d{1,2}\s*(pro|plus...)\b` 中有強制的「英文單字斷字邊界（`\b`）」。當客人打 `17PROMAX` 時，守門員看到 `PRO` 後面跟著 `MAX`（都是英文字母），認定沒有斷字邊界，因此**直接拒絕將其判定為產品查詢**，導致完美重寫的底層搜尋引擎根本沒有被觸發！
- **修復方式**：將縮寫組合 `promax` 與 `pm` 正式納入守門員的合法詞庫白名單，讓這些死忠連排的字元能順利通過意圖過濾器，正確進入底層搜尋程序。

## [v0.5.7] - 2026-03-31
### ♻️ 重構：搜尋引擎分析器（Tokenizer）完整乾淨重寫
- **修復方式**：將原本縫縫補補的分析邏輯徹底打掉重練，改採 6 步驟流水線架構（同義詞展開 → 前處理白名單剝離數字字母 → 斷詞 → Token Group 群組構建 → AND of ORs 資料庫查詢 → 結果評分找尋最精準匹配），全面解決所有怪異連字或多語種別名查詢問題。

## [v0.5.6]
### 🐛 修正：型號縮寫與單字母查詢失敗（如 小米17U, 16PM）
- **根本原因**：上一版（v0.5.4）加入的斷詞會把 `17U` 拆成 `17` 跟 `U`。但系統有一套保護機制定調「單一英文字母可能是噪音字（如 a, s）會被直接略過」。結果 `U` 被略過了，導致只搜尋 `小米 17`，找出了太多不相干款式；再加上資料庫裡其實叫做 `Ultra` 而不是 `U`。
- **修復方式**：
  1. 強化正則表達式，自動拆解所有「數字＋結尾英文字」（如 `S24Ultra` → `S24 Ultra`）。
  2. 新增防呆詞庫（同義詞轉換），將 `u` 轉換為 `ultra`，將 `pm` 轉換為 `pro max` 或 `promax`。
  3. 特別豁免 `u`，讓它不會受到「單字母刪除」規則的阻擋，可以順利進入資料庫成為比對條件。

## [v0.5.5]
### 🐛 修正：禁止 AI 產品回覆中出現報價
- **根本原因**：雖然知識庫規則已註明「不主動報價」，但 `product.service.ts` 的搜尋結果依然將 `NT$ 價格` 直接塞進 AI 的候選 Prompt 裡面，導致 AI 「看見」價錢後認為這是必須告知客人的重要資訊，從而違背了不報價規則。
- **修復方式**：從產品索引注入給 AI 的參考上下文 (`formatProductsAsAiContext`) 以及備用的純文字回覆 (`formatProducts`) 中徹底移除價格欄位，並強制加上第四條回覆規則：「`4. 絕對不可在回覆中列出商品價格或主動報價。`」，以徹底杜絕此現象。

## [v0.5.4]
### 🐛 修正：型號連字搜尋失敗（如 17PRO, iphone16）
- **根本原因**：客戶經常輸入連字型號（如「17PRO」、「iphone16」），系統原本解析時會把它當成「一整個單字」，導致資料庫用 `17PRO` 去比對原本有空格的商品名稱（如 `17 Pro`）時完全命中失敗，AI 因此宣稱找不到。
- **修復方式**：加入智慧斷詞前處理，自動辨識常見型號關鍵字（iphone, pro, max, ultra, plus, mini 等），在解析前自動補上空格（如 `17PRO` 轉為 `17 PRO`，`iphone16` 轉為 `iphone 16`），完美相容上一版的 AND 精準搜尋策略。

## [v0.5.3]
### 🐛 修正：產品搜尋引擎「精準度」升級
- **根本原因**：上一版實作的 OR 策略過於寬鬆（例如搜尋「小米 17」，會撈出所有名字裡有「小米」或「xiaomi」或「17」的商品）。這導致資料庫直接吐回它找到的「前 15 筆」小米商品，結果真正的「小米 17」因為排在第 300 筆而根本進不到 AI 的候選清單裡，使得 AI 誤以為找不到商品。
- **修復方式**：將搜尋策略從單一的大型 OR 改為 **「群組化精準過濾（AND of ORs）」**。現在當客戶輸入「小米 17」時，系統會強制要求資料庫：必須同時包含（小米/xiaomi/redmi/mi 任一字）**「而且」**必須包含（17）。這樣就能濾掉不相干的其他型號，確保 AI 永遠能拿到最精準的商品連結作答。

## [v0.5.2]
### 🐛 修正：知識庫上傳失敗（Extracted content is empty or unreadable）
- **根本原因**：Windows「記事本」預設將 .txt 儲存為 **UTF-16 LE** 編碼（含 `FF FE` BOM），後端直接用 `buffer.toString('utf-8')` 解析導致內容全是亂碼，`content.trim()` 為空，才觸發此錯誤。
- **修復方式**：上傳路由 `/knowledge/upload` 新增智慧編碼偵測：自動識別 **UTF-16 LE BOM**（`FF FE`）用 `utf16le` 解碼；識別 **UTF-8 BOM**（`EF BB BF`）去除 BOM 再解碼；其餘一律以標準 UTF-8 處理。

## [v0.5.1]
### 🐛 修正：產品搜尋引擎升級 & AI 連結格式修正
